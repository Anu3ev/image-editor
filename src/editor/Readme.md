# Архитектура менеджеров редактора

Менеджер остаётся публичной точкой входа для своей области редактора. Внутренние файлы должны называться по своей роли: create, domain, layout, mutation, scaling, editing, event или lifecycle.

Код в файле не должен быть больше 1500 строк. Вынос отдельного кода должен происходить по принципу, когда у кода есть самостоятельная ответственность и после выноса основной сценарий читается проще.

## Корневое правило

В корне папки менеджера остаются только публичный вход и общие типы:

```text
src/editor/shape-manager/
  index.ts
  types.ts
```

Не оставляем в корне файлы-обёртки с re-export на два-три ряда. Если код переехал в папку по роли, импорты должны указывать на эту папку напрямую.

## Роли папок

Для большого менеджера используем такую структуру:

```text
domain/     доменная форма объекта, поиск target, внутренние узлы, metadata, runtime-инварианты
creation/   создание готовой композиции и её частей
layout/     размеры, отступы, измерение текста и применение layout
mutation/   публичные изменения объекта, update pipeline, commit и rehydration
scaling/    обработка scale-событий, live-состояние, ограничения, preview и commit
editing/    редактирование текста или другого внутреннего содержимого
events/     подписки на canvas events и маршрутизация этих событий
lifecycle/  события жизненного цикла объекта и snapshot payload
```

Папки `helpers`, `common`, `internal` и похожие контейнеры не используем. Если хочется назвать файл общим словом, сначала нужно найти его реальную роль.

## Что остаётся в manager

В `index.ts` остаётся публичный API и сборка внутренних частей:

- создание, обновление, удаление и чтение состояния;
- связывание контроллеров и расчётных модулей;
- общий порядок добавления на canvas, выбора объекта, сохранения history и отправки событий;
- совместимость публичного API, который уже используют внешние части проекта.

Менеджер не должен знать все мелкие шаги создания shape, пересчёта текста или применения update. Он вызывает один понятный шаг и сохраняет внешний контракт.

## Как выбирать имя файла

Имя должно объяснять, почему файл существует:

- `factory` — создаёт один объект или узел;
- `builder` — собирает готовую композицию из нескольких частей;
- `pipeline` — последовательно собирает данные для сложного изменения;
- `commit` или `apply` — фактически применяет уже подготовленный результат;
- `layout` — считает или применяет размеры и размещение;
- `measurement` — измеряет текст или геометрию;
- `reference` — находит объект по target, active object или id;
- `nodes` — работает с внутренними узлами композиции;
- `metadata` — читает и записывает persisted metadata;
- `runtime` — восстанавливает временные Fabric-инварианты после create, clone или deserialize;
- `controller` — управляет отдельной областью поведения.

Если класс хранит временное состояние одного пользовательского действия, лучше назвать его по этому действию. Например, для drag/resize/scale-сценария может подойти `Session` или отдельный controller, если он обрабатывает события и завершает действие.

## State и lifecycle

В каждом изменении нужно отдельно держать в голове:

- persisted state — то, что сохраняется в объекте и истории;
- derived state — то, что вычисляется из layout или размеров;
- transient state — временное состояние текущего взаимодействия.

Временное состояние нельзя записывать в persisted metadata, если это не часть доменной модели. Если shape создаётся напрямую, клонируется, восстанавливается из history или приходит из шаблона, один и тот же контракт должен восстанавливаться одинаково.

## Когда делить файл

Если файл вырос больше 1000 строк, это сигнал проверить ответственность. Если файл достиг 1500 строк, его нельзя дальше править без структурного разбиения.

Перед выносом нужно ответить на три вопроса:

- какая у нового файла ответственность;
- станет ли вызывающий код проще;
- не появляется ли новый wrapper, который просто прячет один вызов.

## Проверка после разбиения

После структурного изменения проверяем:

- в корне менеджера нет re-export обёрток;
- у каждого файла одна причина для изменения;
- публичный API остался в manager-е;
- create, update, restore, copy и history не трактуют shape по-разному;
- комментарии описывают реальный контракт;
- в diff нет лишних wrapper-ов, сложных типов ради TypeScript и случайных `as unknown as`.
